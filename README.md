# AI-Generated Text Detector

Estimates whether a piece of text was written by a human or generated by an
LLM, using classical NLP feature engineering rather than a black-box neural
classifier — and shows its work: a sentence-level breakdown highlighting
which parts of the text read as most machine-typical, plus the full
diagnostic feature readout behind every verdict.

```
"Furthermore, it's important to note that this approach offers..."
                                                ▲
                                    flagged: high AI-likelihood
```

## Why classical features instead of a transformer?

Most "AI text detectors" either call a third-party LLM API or fine-tune a
transformer. This project deliberately does neither, for two reasons:

1. **Explainability.** A feature vector of named, human-readable signals
   (perplexity, burstiness, lexical diversity, sentence-length variance,
   POS-tag entropy, readability...) can be shown to the user with a reason
   attached. A 1536-dim embedding can't.
2. **Deployability.** No GPU, no `torch`, no multi-GB model download — the
   whole thing runs on a free-tier CPU instance.

The tradeoff is ceiling accuracy on genuinely subtle, well-edited AI text.
See **Limitations** below — this is a deliberate, documented tradeoff, not
an oversight.

## How it works

1. **Reference language model** — a bigram model (Laplace smoothing) is
   built from a held-out 70% slice of NLTK's Brown corpus. It's used to
   score how "predictable" a piece of text is under typical English word-pair
   statistics — a lightweight proxy for the perplexity signal tools like
   GPTZero compute with a full neural LM, without needing one.
2. **Feature extraction** (`features.py`) — for any text, computes 11
   features: perplexity, *burstiness* (variance of per-sentence perplexity —
   AI text tends to be more uniform sentence-to-sentence), lexical diversity
   (type-token ratio), sentence-length stats, stopword ratio, punctuation
   density, bigram repetition rate, POS-tag entropy, and Flesch readability.
3. **Classifier** (`train.py`) — Logistic Regression, Random Forest, and
   Linear SVM are trained on the feature vectors and compared; the best
   performer (by held-out AUC) is saved.
4. **Inference** (`inference.py` / `app.py`) — a FastAPI endpoint runs the
   same feature pipeline on the full document *and* on each individual
   sentence, so the frontend can highlight specific sentences rather than
   only giving a single document-level score.

## Project structure

```
backend/
├── app.py              FastAPI app (/analyze, /health, /model-info)
├── inference.py         loads the trained model, scores text + sentences
├── features.py          the 11-feature extraction pipeline
├── reference_lm.py       bigram LM (held-out Brown corpus split)
├── train.py              trains + compares LR / RF / LinearSVC, saves the best
├── data/
│   ├── ai_samples.py      synthetic AI-style paragraph generator (see Limitations)
│   └── build_dataset.py    assembles the balanced human/AI training CSV
├── model/                trained artifacts (generated, gitignored)
├── requirements.txt
└── Dockerfile             ready for Hugging Face Spaces (Docker SDK)
frontend/
└── index.html            standalone vanilla HTML/CSS/JS UI, no build step
```

## Running it locally

```bash
cd backend
pip install -r requirements.txt

# one-time setup: builds the training set, the reference LM, and the classifier
python data/build_dataset.py
python -c "from reference_lm import build_reference_lm, save_reference_lm; save_reference_lm(build_reference_lm())"
python train.py

uvicorn app:app --reload
```

Then open `frontend/index.html` directly in a browser (it talks to
`http://localhost:8000` by default — set `window.AI_DETECTOR_API_BASE`
before the script tag to point it elsewhere).

## Deploying

- **Backend** → Hugging Face Spaces, Docker SDK. Push the contents of
  `backend/` as the root of the Space; the included `Dockerfile` builds the
  dataset, trains the model, and starts the API automatically — no manual
  training step needed on the Space itself.
- **Frontend** → any static host (Vercel, GitHub Pages, Netlify). Set
  `window.AI_DETECTOR_API_BASE` to your deployed Space URL.

## API

`POST /analyze`
```json
{ "text": "..." }
```
→
```json
{
  "label": "likely_ai | likely_human | mixed",
  "ai_probability": 0.0,
  "confidence": 0.0,
  "features": { "perplexity": 0.0, "...": "..." },
  "sentences": [{ "text": "...", "ai_probability": 0.0 }]
}
```

## Limitations (read this before trusting the numbers)

This is a portfolio/demo project and is upfront about where it falls short:

- **No real labeled dataset.** There's no network access to a genuine
  human-vs-AI corpus (Hugging Face Hub, Kaggle, OpenAI API) in the
  environment this was built in. The "AI" class is a **template-driven
  synthetic generator** (`ai_samples.py`) that mixes two stylistic
  registers — hedging/listy self-help phrasing, and dense tech-jargon
  prose ("leverage," "synthesize," "unprecedented," "algorithmically") —
  to approximate the range of common LLM stylistic tells. It is *not* real
  LLM output. The "human" class is real text, but pulled from NLTK's Brown
  corpus (1960s news/fiction/essays), not modern blog-style writing.
- **This means the two classes differ by era and register, not just
  human-vs-AI** — a confound the held-out test accuracy (~99–100%) doesn't
  reflect, since train/test are drawn from the same synthetic distribution.
  Real-world accuracy on genuine modern human writing vs. genuine LLM output
  has **not** been formally validated and is almost certainly lower than the
  reported test metrics.
- **Known precision/recall tradeoff:** adding the tech-jargon register
  (to catch genuinely fluent modern AI prose, which the original hedge-y-only
  generator missed entirely) measurably improved AI-text recall, but dense,
  formal *human* writing — corporate reports, academic abstracts — shares
  enough surface features (long sentences, low burstiness,
  nominalization-heavy phrasing) that it can occasionally get flagged as AI
  too. This is an inherent overlap in the feature space, not a separate
  defect.
- **Sentence-level scores are noisy for short sentences** — perplexity and
  POS-tag entropy need a reasonable amount of text to be stable; a 4-word
  sentence can get an unreliable score.
- **Residual sensitivity in the hedge-style register:** a rephrased hedge-style
  AI paragraph without strong tech-jargon markers can still under-score. The
  tech-jargon register added in this revision improved recall specifically
  for fluent, jargon-dense AI prose; other AI stylistic registers not yet
  represented in the generator may still be missed.

**To make this production-grade:** swap `data/build_dataset.py`'s AI-side
generator for a real paired dataset — the
[HC3 dataset](https://github.com/Hello-SimpleAI/chatgpt-comparison-detection)
or Kaggle's "LLM - Detect AI Generated Text" competition data are good
starting points — and retrain with `train.py` unchanged (it's
dataset-agnostic; it only expects a `text,label` CSV).
